AIMS Feature Specifications — Non-Technical Guide

This document describes how each feature would work from the perspective of someone using or managing the system. It is written for people familiar with aid effectiveness, ODA, and development finance but not with software development. For each feature you will find: what it is, why it matters, what users would see and do, what data would be captured or shown, and how it fits with the rest of AIMS.

1. Complete IATI XML Export

What it is

Today, when you “Export to IATI XML” for an activity, the system produces a file that contains the activity’s basic details, participating organisations, and transactions. That file is incomplete by IATI’s full standard: it does not yet include budgets, planned disbursements, policy markers (e.g. gender, climate), tags, the results framework, conditions, contact information, document links, recipient countries and sectors at activity level, forward spending survey (FSS) data, or loan terms and loan status (CRS-add). Many of these are already stored and edited in AIMS; they are simply not yet written into the export file.

Why it matters

  • IATI compliance and validation: A complete export allows the file to pass IATI validation and be accepted by the IATI Registry and by partners who consume IATI data.
  • Transparency and reporting: Donors and countries that publish to IATI need the full picture (financial plans, results, policy markers, documents) in the export, not only in the system.

How we would build it

  • Data already in AIMS: We would connect the export process to every relevant part of the activity that already exists: budgets, planned disbursements, FSS forecasts, policy markers, tags, results and indicators, document links, recipient countries and sectors, financing terms and loan status, and (where applicable) conditions and contact info.
  • One export, one file: When a user clicks “Export to IATI XML,” the system would gather all of that information for that activity and produce a single IATI 2.03–compliant XML file that includes every supported element.
  • Optional check before export: We could add a “Check before export” step that runs the file against IATI’s rules and shows a short report so users can fix issues in AIMS before publishing.

Features included

  • Export includes: activity identity and titles/descriptions; participating organisations (with roles and CRS channel where used); activity status and dates; recipient countries and regions; sectors; budgets; planned disbursements; forward spending survey; transactions; policy markers; tags; results and indicators; document links; conditions; contact info; loan terms and yearly loan status; humanitarian flag and scope where used.
  • Optional pre-export validation report. Filename and content follow IATI 2.03 for Registry or other tools.

2. OECD DAC CRS Reporting Module

What it is

The Creditor Reporting System (CRS) is how DAC members and many other reporters send detailed ODA data to the OECD. Reports are structured into standard tables: Table 1 (ODA by recipient country), Table 2 (ODA by sector), and Table 5 (ODA by channel/modality). Today AIMS holds much of the underlying data but there is no dedicated CRS report to generate these tables or the full CRS++ detail file.

Why it matters

  • Reporting to the OECD: Countries and agencies that report to the DAC need Table 1, Table 2, Table 5, and often the full CRS++ submission. Doing this manually is slow and error-prone.
  • Consistency: Building these reports from the same data that drives IATI and internal dashboards keeps figures consistent.

How we would build it

  • Same data, new reports: Use what AIMS already has: which activities go to which recipient countries, which DAC sectors they use, what amounts were committed and disbursed, and (where recorded) channel codes and flow/finance types. The system would aggregate by the dimensions required for each table.
  • Filters: Users would choose a time period and optionally filter by organisation, country, or sector—same as existing AIMS reports.
  • Three main tables plus detail: Table 1 (ODA by recipient); Table 2 (ODA by sector); Table 5 (ODA by modality/channel); and a full CRS++ row-by-row export for OECD submission.
  • A new section on the existing Reports page, e.g. “CRS Reports,” with one option per table plus “Full CRS++ export,” with “Run report” and “Download” (Excel or CSV).

Features included

  • Report “Table 1 — ODA by recipient,” “Table 2 — ODA by sector,” “Table 5 — ODA by modality/channel” with period and optional filters; downloadable table.
  • “Full CRS++ export” for OECD submission. All figures from the same activity and transaction data; optional short guidance on the Reports page.

3. Forward Spending Plan / MTEF Integration

What it is

Forward spending is the multi-year view of expected aid flows. AIMS already has a Forward Spend tab per activity. What is missing is a portfolio-level view: a single place where government or donors can see total expected flows by year, and optionally by sector, region, or donor, for Medium-Term Expenditure Framework (MTEF) and budget planning.

Why it matters

  • Budget planning: Finance and line ministries need to know how much aid is expected in the next 3–5 years.
  • Predictability: Forward spending visibility supports aid predictability (e.g. GPEDC Indicator 5) and dialogue.

How we would build it

  • No new data entry: Use existing FSS and planned disbursement data; add a “Forward Spending” or “MTEF” view that aggregates across activities.
  • What users would see: By year (total expected disbursements for current and next several years); by sector; by region (if AIMS has location data); optionally by donor. A dedicated page with charts (e.g. bar by year, stacked by sector) and tables; switch between chart and table; download for MTEF tools.

Features included

  • One consolidated view of forward spending (FSS + planned disbursements) for the whole portfolio or filtered subset. Totals by future year with optional breakdown by sector, region, and donor. Charts and tables; option to download data. No duplicate data entry.

4. Counterpart / Co-Financing Tracking

What it is

Counterpart funding (or co-financing) is the government’s own commitment to fund part of a project alongside donor money. Many recipient-country AIMS need to record and monitor these: how much was committed, by which ministry or department, and (optionally) how much has been disbursed. Today AIMS does not have a dedicated place for this.

Why it matters

  • National ownership and accountability: Tracking counterpart funding shows how seriously government is backing each programme.
  • Donor dialogue: Donors often want to see that government is co-investing.

How we would build it

  • For each activity, add a Counterpart funding (or Co-financing) section. Users record one or more counterpart commitments (e.g. ministry, amount, fiscal year, date committed; optional disbursed amount). New tab under “Funding & delivery” next to Budgets and Planned disbursements.
  • Table of commitments with add/edit/remove; optional conversion to a standard currency (e.g. USD). Optional report: total counterpart commitments by year or by ministry across activities.

Features included

  • Per-activity list of counterpart (government) commitments: amount, currency, fiscal year, committing ministry/department, date, optional notes. Optional tracking of “disbursed so far.” Add, edit, delete; optional USD conversion. Optional portfolio-level summary by year or ministry.

5. Disbursement Forecasting / Predictability Analysis

What it is

Predictability here means: how much of what was planned to be disbursed actually was disbursed. AIMS already has screens that compare planned vs actual. We would add a clear predictability metric (e.g. in the spirit of GPEDC Indicator 5b) and a dedicated place to see it (e.g. “In 2024, 78% of planned disbursements were delivered”), plus which activities or donors are above or below that average.

Why it matters

  • Aid effectiveness: Predictability is a core Busan/GPEDC commitment.
  • Planning: Knowing how reliable past plans were helps improve future planning and dialogue.

How we would build it

  • Fix a simple rule, e.g. for a given period compare total actual disbursements to total planned; the ratio is predictability. Using existing data, the system would sum planned and actual amounts (with filters), compute the ratio, and optionally break down by activity or donor.
  • A dedicated Predictability section (e.g. Aid Effectiveness or Analytics): headline number, time series, table (activity or donor, planned, actual, ratio, “on track” / “below plan”), and option to download for GPEDC or other reports.

Features included

  • One clear predictability metric at portfolio level with documented definition. Time series and optional breakdown by sector or donor. Table of activities (or donors) with planned vs actual and ratio. Download of data. All figures from existing planned disbursement and transaction data.

6. Loan Lifecycle Management

What it is

AIMS already stores loan terms and loan status per year for loan activities. What is missing is a lifecycle view: an amortisation schedule, the evolution of outstanding balance over time, and (for ODA reporting) concessionality / grant equivalent.

Why it matters

  • ODA and DAC: Since 2018, DAC reports ODA loans in grant-equivalent terms.
  • Debt management: Finance ministries need repayment schedules and outstanding balances.

How we would build it

  • Amortisation schedule: From existing loan terms, the system would calculate a schedule (principal and interest per period, outstanding balance); users see it as a table and optionally chart. Where actual loan status exists, show “planned” and “actual” side by side.
  • Outstanding balance over time: Table or chart of balance year by year.
  • Concessionality / grant equivalent: A calculation (DAC methodology) using the loan’s rate and repayment profile and a reference rate; show “Grant equivalent: $X” on the activity and use in reports. Document methodology.
  • All in or next to the existing Loan terms / Financing terms area. Optional report: “All loan activities with outstanding balance and grant equivalent.”

Features included

  • Amortisation schedule from loan terms (table and optional chart). View of outstanding balance over time. Concessionality / grant-equivalent calculation per loan with documented methodology. Display on activity and use in ODA/CRS-style reports. Optional portfolio-level list or export.

7. Multi-Donor Trust Fund / Pooled Fund Management

What it is

A pooled fund receives money from several donors and disburses to downstream projects. Today AIMS can link activities but does not aggregate the financial picture: who gave how much to the fund, and how much the fund passed on to which sectors or regions.

Why it matters

  • Accountability: Fund managers need to report to contributors and to government.
  • Planning: Government needs to see flows through funds by sector/region.

How we would build it

  • Use existing linking (e.g. parent/child activities). A pooled fund is an activity marked as “fund” with child activities. Use same transactions and planned disbursements, interpreted as (1) money into the fund and (2) money out to child activities.
  • For a fund activity: Contributions view (by donor organisation, amounts, optional by year); Disbursements view (by child activity, optionally by sector/region). Optional Pooled funds list/dashboard with total contributions and disbursements per fund.

Features included

  • Mark activity as “pooled fund” and link child activities. Per fund: Contributions view and Disbursements view. Optional Pooled funds overview. All figures from existing transactions, planned disbursements, and activity links.

8. Donor Division of Labour / Fragmentation Analysis

What it is

Fragmentation means many donors in the same sector or region. Division of labour is donors and government agreeing who does what. AIMS already has fragmentation views (e.g. heatmap donors × sectors). We would add a clearer division of labour angle: per sector (or region), how many donors, how concentrated funding is (e.g. concentration index), and optionally comparison to an agreed “ideal.”

Why it matters

  • Paris/Busan: Reducing fragmentation and improving division of labour is a long-standing commitment.
  • Dialogue: A “sector X has 12 donors, sector Y has 3” view supports rebalancing discussions.

How we would build it

  • Use the same data as fragmentation heatmaps. Add per sector (and optionally region): number of donors, total funding, share per donor, and a concentration index (e.g. Herfindahl). No new data entry.
  • A Division of labour (or “Fragmentation summary”) view: table (one row per sector: number of donors, total amount, top 3 donors, concentration index); optional chart; same filters as elsewhere; download for strategy and dialogue. Optional later: column comparing actual vs recommended division of labour if configured.

Features included

  • Per sector (and optionally region): number of donors, total funding, top donors and shares, concentration index. Single view with filters and download. Optional later: comparison to agreed “ideal” if configured.

9. Excel/CSV Template-Based Bulk Data Entry

What it is

Today bulk data often enters via IATI XML import. Many partners prefer to submit in Excel or CSV using a fixed template. This feature would provide: (1) downloadable templates (e.g. Activities, Transactions, Budgets) with clear columns and instructions, and (2) an upload and check process: partners upload, the system checks (required fields, valid codes) and either imports or shows errors so they can fix and re-upload.

Why it matters

  • Inclusivity: Not all partners use IATI; Excel is universal.
  • Quality: Standard template and automated checks reduce missing or invalid data.
  • Efficiency: Bulk upload is faster than entering many items one by one.

How we would build it

  • Templates: Define Excel (or CSV) templates with fixed columns and an instruction sheet (required fields, codes). Users download from a “Bulk upload” or “Templates” page.
  • Upload: User selects template type, uploads file. System checks every row (e.g. Activity ID filled, sector valid DAC code, date format). User sees: how many valid rows, list of errors (row, column, message). They fix and re-upload or “Import only valid rows.”
  • Import: On confirm, system creates/updates activities, transactions, or budgets using the same validation as the rest of AIMS. Summary (“80 activities created,” “2 skipped”) and link to view imported items. Governance: who can use bulk upload; “new only” vs “update existing” per template.

Features included

  • Downloadable templates for at least Activities, Transactions, Budgets (and optionally Planned disbursements, Participating organisations) with instruction sheet. Upload page with automatic check and clear error report. Option to import only valid rows with summary. Consistency with existing AIMS rules.

10. Procurement / Contract Tracking

What it is

Many activities involve procurement: tenders, contracts, and contractors. Today AIMS does not record these. This feature would add a simple way to record per activity: tenders (reference, title, procedure type, deadline, status) and contracts (reference, title, contractor, value, start/end date, status) to support implementation monitoring and accountability.

Why it matters

  • Implementation oversight: Project managers and government need to see how procurement is progressing.
  • Transparency: “Who got the contract?” can be answered in one place with the activity.

How we would build it

  • Each activity gets an optional Procurement (or Contracts) section with two lists: Tenders and Contracts. Tenders: reference, title, procedure type, status, deadline, notes; add/edit/remove. Contracts: reference, title, contractor (from organisation list or text), value, currency, start/end date, status, notes; add/edit/remove.
  • Optional report: “Total contract value by activity” or “Contracts by contractor” with filters. Same look and feel as other AIMS sections (tables, add/edit buttons).

Features included

  • Per-activity Tenders list and Contracts list with add, edit, delete. Optional portfolio-level summary (total contract value, by contractor or activity) with filters.

11. Climate Finance Tagging and Reporting

What it is

Climate finance is ODA that targets climate mitigation or adaptation. AIMS already has policy markers (e.g. Rio Markers) including climate; what is missing is a dedicated climate finance view: one place to see total climate-tagged funding over time, by sector, by donor, and optionally adaptation vs mitigation.

Why it matters

  • Reporting: Countries and donors report climate finance to UNFCCC, OECD, and others.
  • $100B and beyond: Tracking progress toward climate finance goals requires consistent tagging and aggregation.

How we would build it

  • Use existing policy marker for climate (e.g. Rio Marker); activities/transactions with that marker count as “climate finance.” No new field from scratch. If adaptation vs mitigation is stored, use it; otherwise start with “total climate.”
  • A new Climate finance section (Analytics or Aid Effectiveness): over time (chart and table); by sector; by donor; adaptation vs mitigation if data allows; optional target progress. Export for UNFCCC, OECD, or national reports. Document definitions.

Features included

  • Definition based on existing policy markers; no duplicate tagging. Climate finance dashboard: totals over time, by sector, by donor, adaptation vs mitigation if data allows. Charts and tables; download. Optional progress toward climate finance target. Same filters as rest of AIMS.

12. Environmental and Social Safeguards Tracking

What it is

E&S safeguards are assessments and measures to avoid or mitigate harm (e.g. environmental impact assessments, grievance mechanisms). Some donors and governments require a record per activity: type of safeguard, whether assessment was done, status (e.g. planned, in progress, completed), and optionally a link to the document. Today AIMS does not capture this.

Why it matters

  • Compliance: Donors and regulations often require evidence that safeguards have been considered and implemented.
  • Risk management: A central view of “which activities have completed E&S assessments” helps track gaps.

How we would build it

  • Each activity gets an optional Safeguards (or E&S safeguards) section. Users record one or more safeguard records: type (e.g. environmental assessment, social assessment, grievance mechanism), status (not applicable, planned, in progress, completed), date, notes, optional document link. Table with add/edit/remove.
  • Optional portfolio view: “Activities with incomplete safeguards” or “Safeguards by type and status” with filters. Types could be configurable to align with country or donor taxonomies.

Features included

  • Per-activity Safeguards section: list of records (type, status, date, notes, optional document link). Configurable types; status options. Add, edit, delete; optional portfolio-level view with filters.

13. Results Aggregation and Portfolio-Level Reporting

What it is

AIMS already has a results framework per activity (results and indicators with targets and actuals). What is missing is a portfolio-level view: across many activities (e.g. all health sector), “how many clinics were built?” or “what is the aggregate progress on indicator X?” This feature would add reports that aggregate results and indicators across activities by sector, country, or other dimension.

Why it matters

  • Sector reporting: Sectors need to report “across all projects we achieved X.”
  • SDG and national targets: Linking aggregated results to national or global indicators requires consistent summing or averaging.

How we would build it

  • Use existing result and indicator data. Users choose a filter (e.g. sector “Health,” region, organisation) and the system aggregates: by result type; by indicator (sum or average actuals where comparable); by sector or country. Document aggregation rules (e.g. sum for count, average for percentage).
  • A new Portfolio results (or Results overview) page: filter choice; tables (e.g. “Results by sector,” “Indicators by type”); optional charts; download for sector or SDG reports. No change to activity-level results—only a layer of aggregation on top.

Features included

  • Portfolio results view with filters. Tables by sector and by indicator; clear rules for sum vs average and short documentation. Optional charts and download. All data from existing activity results and indicators.

14. Comprehensive Audit Log with Search and Export

What it is

AIMS already keeps an activity log (who did what and when). What is missing is search (e.g. “all changes to activity A in the last month,” “all actions by user B”), filter by type of action or date range, and export for auditors. Some users also want to see what changed (e.g. “Budget changed from $1m to $1.2m”) not just “Budget was edited.”

Why it matters

  • Accountability: Auditors and oversight need to trace who changed what and when.
  • Compliance: Some regulations require keeping and producing an audit trail.

How we would build it

  • Search and filters: Add to the existing Activity log / Audit log page: date range, type of action, who (user), which entity (e.g. activity); and a search box for free text in the log description. Results as list/table, most recent first.
  • Export: Button to export the filtered log to Excel or CSV for auditors.
  • What changed (optional): Where we store old/new value for an edit, add a “View details” or “View change” per row (e.g. “Budget: $1,000,000 → $1,200,000”). If we don’t yet store field-level changes, we could add that for key fields in future; phased: first search and export, then field-level diff where feasible.
  • Document retention and who can access the log.

Features included

  • Filters (date range, action type, user, entity) and search in log descriptions. Export of filtered log to Excel or CSV. Optional “View change” for edit actions where old/new value is stored. Documentation on retention and access.

15. Multi-Language User Interface (i18n)

What it is

Today the AIMS interface (menus, buttons, labels, messages) is in English only. Multi-language support means users (or an administrator) choose a preferred language and every label, button, and message is shown in that language. The data (activity titles, descriptions) can already be in any language; this feature is only about the interface language.

Why it matters

  • Inclusivity: National staff and partners often prefer to work in the national language.
  • Uptake: Lower barrier when the tool “speaks” the user’s language.
  • Compliance: Some countries require government systems in the official language(s).

How we would build it

  • Choice of language: Add a language selector (header or profile): dropdown of supported languages (e.g. English, French, Spanish). On selection, the entire interface switches; choice is remembered (e.g. in user profile or browser).
  • What gets translated: Every fixed text: navigation, page titles, form labels, table headers, buttons, validation messages, report names. We would not translate user-entered content (titles, descriptions)—only the “frame” of the application.
  • Phased rollout: Start with most used parts (main menu, Dashboard, Activities list and detail, Reports); then extend to Organisations, Data Clinic, Admin. Show “This page is available in English and French” until all pages are done. Translations from professional or bilingual staff; if a translation is missing, fall back to English. Same term translated the same way everywhere for consistency and manuals.

Features included

  • Language selector in header or profile; choice remembered. Full translation of interface text for at least two languages, option to add more. Phased rollout with indication where translation is complete. No translation of user content; fallback to default language; consistent terminology.

16. Offline Data Entry / Progressive Web App (PWA)

What it is

Today AIMS requires an internet connection. Offline support would let users open AIMS and enter or edit data when offline; the system would store that work locally and send it to the server when connection is back (e.g. complete a monitoring form in a village with no signal, sync next day in the office).

Why it matters

  • Field use: Monitoring and partner meetings often happen where internet is unreliable.
  • Resilience: When network or power is down, work is not lost if saved locally first.

How we would build it

  • What works offline: A subset of AIMS—e.g. view and edit one activity (the one “prepared for offline”), add a transaction or result, or add a new activity with basic fields. When online, user “prepares for offline” (downloads that activity and code lists); when offline, data is available from the device and they can edit/add within that scope. Full search might stay online-only.
  • Saving offline: “Save” while offline stores the change in a local queue; message: “Saved locally. Will sync when you’re back online.” Optional indicator: “3 changes pending sync.”
  • Sync when online: System automatically sends queued changes. If server accepts, queue is cleared and user sees “Synced.” If server rejects (e.g. someone else changed the same activity), show “1 change could not be synced: conflict on Activity X” and let user choose (e.g. “Use my version” or “Use server version” or “Edit again”). No data loss.
  • Document limits (e.g. only a few activities prepared; no real-time collaboration; large reports online-only). Data on device protected (e.g. only when logged in; cleared on logout or after inactivity).

Features included

  • Offline preparation of one or a few activities and code lists. Offline editing and “Save” with “Saved locally, will sync later” message. Sync when online; conflict handling. Clear scope of what works offline and what doesn’t; documented. Security: offline data tied to session, cleared on logout or inactivity.

17. South–South and Triangular Cooperation Tracking

What it is

South–South cooperation (SSC) is when one developing country supports another. Triangular cooperation is when a traditional donor supports that. Reporting often treats these differently from traditional North–South ODA. Today AIMS does not explicitly flag whether an activity is SSC or triangular. We would add a single classification per activity (e.g. North–South, South–South, Triangular) so reports and dashboards can show “how much is SSC?” and “how much is triangular?”

Why it matters

  • Reporting: Many countries and agencies report SSC and triangular separately to the UN, OECD, or regional bodies.
  • Policy: National strategies often set targets or priorities for SSC.

How we would build it

  • Add a Cooperation type (or “Modality”) field to the activity: e.g. “North–South (traditional ODA),” “South–South,” “Triangular,” and possibly “Other” or “Not specified.” Users choose one when creating or editing (e.g. in General or Classification). Field optional so existing activities are not forced to be reclassified immediately.
  • Add filters to activity list and key reports (e.g. “Filter by cooperation type: South–South”). Add a dashboard or report card: “Cooperation type breakdown” (totals by type, e.g. North–South: $X, South–South: $Y, Triangular: $Z) with chart and optional download. Same list of options everywhere; optional short guidance (e.g. “South–South = developing country to developing country”) in help text.

Features included

  • Cooperation type field per activity (North–South, South–South, Triangular, optional Other/Not specified); optional. Filter on activity list and reports. Dashboard/report “Cooperation type breakdown” with chart and download. Consistent list and labels; optional help text.

18. ODA Eligibility Checker

What it is

Not all aid counts as ODA. The DAC has rules: recipient on ODA-eligible list, flow concessional enough (grant or grant-equivalent above threshold), purpose development. Today AIMS does not check whether an activity qualifies as ODA. An ODA eligibility checker would run a simple check per activity (or portfolio) and show “Eligible” or “Not eligible” with short reasons (e.g. “Recipient not on ODA list,” “Concessionality below threshold”).

Why it matters

  • Reporting: Countries and agencies that report to the DAC need to know which activities count as ODA before submitting.
  • Quality: Catching ineligible activities early avoids corrections and reputational risk.

How we would build it

  • Rules: (1) Recipient: at least one on current DAC ODA-eligible list (updated periodically). (2) Concessionality: for grants assume eligible; for loans use grant equivalent (from Loan lifecycle feature if implemented) and check DAC threshold; if not calculated, show “Cannot assess—loan terms or grant equivalent missing.” (3) Purpose: activity has development-relevant sector; if none or non-ODA sector, flag “Purpose not clearly development.” Document rules.
  • On each activity, a small ODA eligibility block (e.g. Summary or Classification): Result “Eligible” (green) or “Not eligible” (red/amber) and Reasons (e.g. Recipient: OK. Concessionality: OK. Purpose: OK.).
  • A report “ODA eligibility”: list all activities (or filtered) with Activity ID, Title, ODA eligible, Reasons; optional filter “Show only not eligible”; optional download. Checker does not delete or hide ineligible activities—only flags them (some may be intentionally non-ODA).

Features included

  • Per-activity ODA eligibility (Eligible/Not eligible and reasons) using current DAC rules. Portfolio report with filter “not eligible only” and optional download. Documented rules. No automatic removal of ineligible activities.

19. Debt Sustainability Integration

What it is

Debt sustainability refers to whether a country can service its debt; the IMF and World Bank produce DSAs and assign risk ratings (e.g. low, moderate, high, in debt distress). Today AIMS does not link to this. The feature would show the current DSA rating (and optionally a link to the assessment) for each recipient country, so when users look at loans or activities in that country they see the context (e.g. “This country is in high risk of debt distress”).

Why it matters

  • Loan decisions: Government and donors need to know the country’s debt situation when considering new loans.
  • Reporting: Debt sustainability is often reported alongside aid and loan data.

How we would build it

  • For each country (in country summary or when viewing activities there), show: Debt sustainability: e.g. “Moderate risk (IMF–World Bank DSA, January 2025)” and optionally a link to the full DSA. Data source: (1) Manual: Designated person enters/updates DSA rating and link per country in a “Country settings” or “DSA” list. (2) Automatic (later): If IMF/World Bank provide a public feed, we could pull it periodically. Start with manual.
  • Display on country page and optionally on activity when single recipient country. Optional report or filter: “Activities in countries at high risk of debt distress.” No change to how activities or loans are entered—only extra context (the rating) next to the country.

Features included

  • DSA rating and link shown per recipient country. Data entry/update of rating and link per country (manual, by authorised user), with date and source. Optional automatic update from external DSA feed. Display on country view and optionally activity view; optional report/filter “Activities in high debt-risk countries.” No change to loan or activity data.

20. Outcome-Based / Results-Based Financing (RBF) Tracking

What it is

Results-based financing (RBF) includes payment by results (PbR), development impact bonds (DIBs), outcome funding—often reported separately from standard grants or loans. Today AIMS records finance types and results but has no explicit flag for “this activity is RBF.” Adding a flag would allow filtering and reporting “how much is RBF?” and “which activities use RBF?”

Why it matters

  • Reporting: Donors and countries increasingly report RBF separately in ODA and aid effectiveness reports.
  • Learning: Tracking which activities use RBF supports evaluation and policy dialogue.

How we would build it

  • Add a way to mark an activity (or optionally a transaction) as results-based financing: e.g. a checkbox (“This activity uses results-based financing (e.g. payment by results, DIB)”) or an “RBF type” dropdown. In the activity form, Funding or Classification section, with short help text.
  • Add filter on activity list and reports (“RBF only”). Add dashboard or report card: “RBF volume” (total commitments or disbursements for RBF activities, by year or sector) with chart and optional download. Optional later: short summary “Results linked to this RBF activity” where both RBF and results exist.

Features included

  • RBF flag or type per activity (e.g. checkbox or RBF option); optional short list (PbR, DIB, other). Filter on activity list and reports. Dashboard/report “RBF volume” with chart and download. Optional link to results/indicators for RBF activities. Consistent definition in help text or documentation.

21. Country Strategy / Partnership Framework Management

What it is

Many countries have Country Partnership Frameworks (CPF), Country Strategy Papers (CSP), or Joint Country Strategies that set priorities. Today AIMS does not store these or link activities to them. This feature would: (1) Record each strategy (title, country, type, period, summary, document link), and (2) Link activities to one or more strategies (or to priorities within a strategy), so reports can show “How much aid is aligned to the CPF?” or “Which activities support Priority 1?”

Why it matters

  • Alignment: Government and donors want to see whether aid is aligned with agreed country strategies.
  • Reporting: “Alignment to CPF” is a common ask in aid effectiveness and dialogue.

How we would build it

  • Strategy record: A Strategies (or Partnership frameworks) area where authorised users create and edit strategies: Title, Country, Type (e.g. CPF, CSP, JCS), Period (start/end), Summary, Document link; optionally Priorities (e.g. “Priority 1: Health”).
  • Linking activities: When editing an activity, add “Link to strategy” (or “Alignment”): user selects one or more strategies and optionally which priority. Stored for reporting.
  • Where users see it: Strategies page (list with title, country, type, period); on the activity (“Aligned to: CPF Kenya 2023–2027, Priority 1”); and a report/dashboard “Alignment to CPF” (funding or activity count by priority for a chosen strategy/country) with optional chart and download. Clear roles for who manages strategies vs who links activities.

Features included

  • Strategy register: create and edit strategies (title, country, type, period, summary, document link, optional priorities). Link activities to strategies (and optionally priority) from activity form. Strategies page; optional list of activities per strategy. Alignment report (funding or count by priority or linked vs not linked) with optional chart and download. Clear roles.

22. Parliamentary Reporting Module

What it is

Many countries present aid data to parliament in a standard format: executive summary (totals, by sector, by donor), detailed table of activities, and annex (methodology, data quality). Today AIMS has general reports but no parliament-specific layout or template. This feature would provide one or more report templates for parliamentary use with fixed structure, clear titles, and optional PDF export for printing or formal submission.

Why it matters

  • Accountability: Parliament holds government to account on aid; a standard report format supports that.
  • Efficiency: Producing the report from AIMS avoids rebuilding it in Word or Excel every time.

How we would build it

  • Content: Agreed with user (e.g. Ministry of Finance): (1) Cover and title (e.g. “Aid Report to Parliament, Fiscal Year 2024”), (2) Executive summary (total ODA, top sectors, top donors, key messages), (3) Detail table (activities with Activity ID, Title, Donor, Sector, Commitment, Disbursement, Status), (4) Annex (methodology, data quality). All figures generated from AIMS data.
  • On Reports page, new section “Parliamentary reports” with “Executive summary” and “Full report.” User chooses reporting period and optional filters, clicks “Generate report.” System fills template; result shown on screen (HTML) for review; “Download PDF” produces print-friendly PDF (headers, page numbers, optional TOC). Template design (headings, logo, disclaimer) agreed for formal submission; configurable title for different countries.

Features included

  • Parliamentary report template(s): at least Executive summary + Detail table + Annex, with cover and title. Generate from AIMS (live data); user selects period and optional filters. Preview on screen before download. Download as PDF with print-friendly layout. Configurable title and basic branding.

23. Donor Self-Assessment Portal

What it is

Data Clinic today lets authorised users see data quality issues across the whole database. Donors often want to see their own data quality: “How complete and timely is the data for activities where we are the funder?” A donor self-assessment would give each donor organisation a dedicated view: scorecards (e.g. “Completeness: 85%,” “Timeliness: 90%”) and a list of their activities that need attention, so they can improve data before government validates it, without seeing other donors’ data.

Why it matters

  • Quality: When donors see their own gaps, they fix them faster; reduces validation burden on government.
  • Fairness: Each donor sees only their own data and scores.

How we would build it

  • Self-assessment shows data only for activities where the current user’s organisation is funder or accountable. (1) Scorecards: e.g. Completeness (% with all required fields), Timeliness (% with transaction in last 12 months), Sector coverage (% with at least one sector); optional traffic lights (green/amber/red). (2) List of activities needing attention: e.g. “15 activities with missing sector,” “8 with no transaction in 12 months,” with links to open and fix. Reuse same Data Clinic checks; only filter by organisation and new page are new.
  • “Self-assessment” or “My organisation’s data quality” page (or Dashboard tab for donor users). Government/super users still see full Data Clinic. Optional: monthly snapshot of scores; download of self-assessment report (PDF/Excel) for headquarters or government.

Features included

  • Donor-only view (per organisation). Scorecards (Completeness, Timeliness, Sector coverage; optional traffic lights). List of activities needing attention with links. Reuse of Data Clinic checks; filter by organisation. Optional monthly snapshot and download of self-assessment report.

24. Automated Data Quality Alerts

What it is

Today users must open the Data Clinic to see data quality issues. Automated alerts would proactively notify the right people when quality drops (e.g. “Organisation X has 15 activities with missing sectors” or “Your portfolio has 10 activities with no transaction in 12 months”). Alerts could be sent by email or shown in the application, on a schedule (e.g. every Monday) or when a threshold is crossed.

Why it matters

  • Proactive quality: Problems are fixed sooner when people are reminded automatically.
  • Accountability: Regular alerts create a habit of “we check our data every week.”

How we would build it

  • Rules: e.g. (1) Every Monday run same checks as Data Clinic and for each organisation with issues send an email to that organisation’s contact(s): “You have X activities with missing sector; here is the list.” (2) Or: when issue count for an organisation goes above 10, send alert immediately. (3) Or: every month send government a summary (“Total activities with issues: 50; by organisation: A 10, B 15, …”). Start with one or two rules (e.g. weekly email per organisation).
  • Each organisation has one or more alert recipients (email addresses), configurable by admin or lead user. Email content short and actionable: what is wrong, how many, “Please log in to AIMS and review Data Clinic (or Self-assessment). List of activity IDs: …” Optional link “View full report in AIMS.” Configuration: “Send organisation alerts: Weekly Monday 9am,” “Send summary to government: Monthly 1st”; optional “Alert only when issue count > N.” System runs checks in the background (scheduled task) and sends emails. Optional log of alerts sent to avoid duplicate sends.

Features included

  • Scheduled alerts (weekly or monthly) running same quality checks as Data Clinic. Per-organisation alerts (email or in-app) listing their activities with issues and links to AIMS. Configurable recipients and optional summary alert for government. Optional thresholds and alert history. Short, actionable message content.

25. Historical Exchange Rate Gain/Loss Tracking

What it is

When a commitment (or planned disbursement) is in another currency, AIMS converts it to USD using the rate on a given date. Over time, rates change, so the USD value of that same commitment can change—that is exchange rate gain or loss. Today AIMS does not track this. The feature would re-value commitments (and optionally planned disbursements) at a reporting date and show the difference from the original USD value so finance and audit can see “we had $X gain/loss due to exchange rates” and reconcile with treasury.

Why it matters

  • Financial reporting: Many countries report aid in USD; “exchange rate effect” explains why totals changed.
  • Reconciliation: Treasury and aid units need to reconcile AIMS with other systems; exchange rate gains/losses are often a line item.

How we would build it

  • For each commitment (and optionally planned disbursement) in non-USD: (1) Original USD value (at commitment date—what AIMS already does), (2) USD value at reporting date (using rate on a chosen date, e.g. 31 Dec 2024). Difference = gain or loss per line. Aggregate across all such commitments: total gain, total loss, net. Use same exchange rate source as AIMS (e.g. central bank). User chooses reporting date when running the report.
  • New “Exchange rate impact” (or “Revaluation”) section under Financial Analytics or Reports: table (by period or activity: Original USD, Re-valued USD, Gain/Loss); total line (net gain or loss); optional chart. User selects reporting date and filters; download (Excel) for financial reports or audit. Apply to commitments first; optionally extend to planned disbursements. No change to daily data entry—only an analytical report.

Features included

  • Re-value commitments (and optionally planned disbursements) in non-USD to USD at a user-selected reporting date, using same rate source as AIMS. Calculate gain/loss per line and aggregate (total gain, total loss, net). Report “Exchange rate impact”: table, total net gain/loss, optional chart; filters and download. No change to day-to-day entry; documentation of which rate is used for audit.
