AIMS Frontend UI Consistency Audit

Comprehensive analysis of 1055 TypeScript components across pages, modals, forms, tables, and more

Executive Summary

The AIMS frontend exhibits substantial inconsistency across 15 critical UI dimensions. The codebase shows signs of organic growth without centralized UI governance, resulting in ~80+ user-visible inconsistencies that range from high-impact (modal styling) to low-impact (icon sizing). Most patterns have clear majorities, but a significant minority of components deviate.

Severity & Audit Methodology

High Severity (12): Visual chaos affecting user experience, breaks established brand patterns, affects accessibility or trust (e.g., modal header rule violations, critical button placement inconsistencies).

Medium Severity (28): Noticeable drift from majority pattern, affects multiple pages, confusing but not breaking (e.g., form label styling variations, spacing divergence).

Low Severity (44): Minor deviations in secondary patterns, isolated to few files, visual polish opportunity but not critical (e.g., icon size variations in edge cases).

Methodology: Analyzed 1055 TSX files using regex pattern matching and manual sampling. For each dimension, identified the dominant (majority) pattern, counted deviations, and assigned severity based on user impact.

2. Button Variants & Sizing

Variant Distribution:
• variant="outline": 333 instances (dominant for secondary/tertiary actions)
• variant="ghost": 120 instances (used for light/icon buttons)
• variant="destructive": 5 instances (delete/cancel)
• variant="link": 4 instances
• variant="secondary": 2 instances
• variant="default": 1 instance (rare, should be primary action)
No variant specified: 1095 instances (defaults to primary)

Issues Found:

Issue Count Impact Severity
Missing variant attribute: 1095 buttons rely on implicit default, unclear when override intended 1095 High confusion when reviewing code; maintenance risk if default changes MEDIUM
Destructive variant underused: Only 5 delete/cancel buttons explicitly destructive; most use outline or ghost 150+ Delete actions not visually distinct; users may not perceive risk HIGH
Ghost variant overloaded: Used for both icon buttons AND secondary actions; inconsistent semantics 120 Ambiguous intent; harder to maintain MEDIUM
Outline as de-facto secondary: 333 outline buttons, some used where primary action might be clearer 333 Action hierarchy unclear in some modals/forms LOW

Button Sizing

Size Distribution:
• No size attribute: 5654 instances (defaults to "default")
• size="sm": 1767 instances (compact buttons)
• size="lg": 21 instances (rare; only landing pages, hero sections)
• size="xl": 0 instances

Issue: Extreme bias toward default size; very few large buttons despite multiple sections that could benefit from prominence.

Recommendation: (1) Standardize button ordering: primary action on right, cancel on left in modals/forms. (2) Create visual guide for variant usage: primary/outline/ghost/destructive. (3) Audit 150+ delete buttons to use destructive variant. (4) Consider variant="secondary" alias if distinction from outline is unclear.
